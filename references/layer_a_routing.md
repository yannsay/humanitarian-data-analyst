# Layer A — Routing Guide

This is the detailed reference for **Step 1 — Route** of the pipeline. Read it when
you need more than the procedure summary in `SKILL.md`.

## What Layer A is

A YAML representation of the **HumSet/DEEP humanitarian analytical framework**: the
controlled vocabulary humanitarian analysts already use to classify information. It
has 66 nodes across five axes. Layer A answers one question: *what is this analyst
question about, and is it in scope for the rest of the pipeline?* It does no
analysis itself.

## The file layout

```
ontology/
├── index.yaml          # compact routing surface — ALL 66 nodes, one gist + synonyms each
├── sectors/            # 11 nodes   — "what domain of need?"
├── pillars_1d/         # 6 nodes    — "type of information"
├── subpillars_1d/      # 27 nodes   — children of 1D pillars
├── pillars_2d/         # 6 nodes    — "analytical lens"
└── subpillars_2d/      # 16 nodes   — children of 2D pillars
```

Each axis folder has an `_axis.yaml` (axis-level description) plus one file per
node. A node file contains the full record:

- `description` — what analysts tag here, with concrete examples
- `synonyms` — alternative phrasings an analyst might use
- `example_questions` — questions that should route to this node
- `distinguish_from` — how to tell this node from siblings/cousins it's confused with
- `excerpt_count`, `saturation_batches`, `low_support` — provenance / confidence

## The routing procedure

1. **Read `ontology/index.yaml` first.** Never route from memory of the framework —
   the index is cheap and it is the source of truth. Match the question's topics
   against `gist` and `synonyms`.

2. **Expect multiple axes per question.** The 1D and 2D pillar axes are
   **orthogonal**: a single question can carry both. "What's the status of water
   access in IDP camps?" is:
   - Sector: **WASH**
   - 2D lens: **Humanitarian Conditions → Living Standards** (present-state material conditions)
   It does *not* necessarily carry a 1D tag. Don't force every axis.

3. **Open candidate node files and read `distinguish_from` before committing.** This
   is the step that is most often skipped and most often the source of error. The
   classic confusions:
   - `Humanitarian Conditions → Living Standards` (material conditions people have/lack)
     vs `→ Physical And Mental Well Being` (health/psychosocial outcomes)
     vs `Impact → Impact On People` (same deficits, but attributed to a shock as cause).
   - `Humanitarian Conditions` (present lived state) vs `At Risk` (future/potential
     vulnerability) vs `Impact` (what the shock did).
   - `Agriculture` (production side) vs `Food Security` (consumption/access outcomes)
     vs `Livelihoods` (income-generating capacity).
   - `Cross` is **not** a lazy fallback for anything multi-sector — it is for excerpts
     where the cross-sector entanglement is itself the analytical finding. If you can
     name one primary sector, use it.

4. **Emit the routing table** exactly as specified in `SKILL.md`, then hand off to
   Step 2. Do not begin analysis here.

## Worked example

> **Question:** "How are displaced families coping with the lack of food and where are they getting water?"

Routing:

| Topic                          | Sector        | Pillar (1D / 2D)                      | Subpillar                          |
|--------------------------------|---------------|---------------------------------------|------------------------------------|
| coping with lack of food       | Food Security | Humanitarian Conditions (2D)          | Coping Mechanisms                  |
| where getting water            | WASH          | Humanitarian Conditions (2D)          | Living Standards                   |
| displaced families             | —             | Displacement (1D)                     | Type/Numbers/Movements             |

Note how "coping" pulls **Coping Mechanisms**, not Living Standards — check
`distinguish_from` on both to confirm: Living Standards = what people have/lack;
Coping Mechanisms = the adaptive strategies they use *because* standards fell.

## Confidence flags

Nodes carry `low_support: true` when the underlying HumSet sample was thin. Treat
routing to a low-support node as lower-confidence and say so in the table notes if
it affects the answer.
